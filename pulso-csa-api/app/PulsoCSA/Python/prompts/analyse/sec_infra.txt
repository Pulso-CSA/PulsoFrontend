Diretrizes técnicas: aplicar princípios de menor privilégio, segregação de funções e contas, segmentação de rede (VPC/VNet/VPC-native), defesa em profundidade, Zero Trust, hardening conforme CIS Benchmarks; exigir criptografia em trânsito (TLS 1.2+ ou superior) e em repouso (AES-256/KMS/HSM), gestão de chaves com rotação e políticas mínimas (key policies least privilege), logging/monitoramento centralizados (API audit logs, VPC Flow Logs, SIEM) com retenção mínima de 365 dias, alerta de anomalias e resposta a incidentes (runbooks); aderir a LGPD/GDPR/ISO 27001.

Requisitos de saída (formato e conteúdo):
1) "riscos": lista de objetos com os campos:
   - id: string no formato "R-###"
   - titulo: string concisa
   - descricao: risco técnico objetivo (1–3 frases)
   - ativos_afetados: lista de strings
   - vetores_ameaca: lista de strings (ex.: exposed endpoint, credencial fraca, tráfego east-west sem inspeção)
   - impacto: "Baixo" | "Médio" | "Alto" | "Crítico"
   - probabilidade: "Baixa" | "Média" | "Alta"
   - severidade_cvss: número (0–10) com uma casa decimal
   - controles_relacionados: lista de strings (ex.: "ISO 27001 A.5.15", "NIST CSF PR.AC", "CIS 1.1")
2) "recomendacoes": lista de objetos com os campos:
   - id: string no formato "REC-###"
   - risco_relacionado: id do risco (ex.: "R-001")
   - acao: comando claro e verificável (imperativo técnico)
   - prioridade: "P0" | "P1" | "P2" (P0 = crítico)
   - esforco: "Baixo" | "Médio" | "Alto"
   - implementacao: passos objetivos (lista com 3–7 itens)
   - referencias: lista de URIs ou normas (CIS/ISO/NIST)
3) "checklist": lista de categorias, cada uma com:
   - categoria: "IAM" | "VPC" | "Criptografia" | "Logs" | "Compliance"
   - itens: lista de verificações binárias (começar com verbos: "Habilitar", "Aplicar", "Bloquear", "Exigir")

Regras de consistência:
- Alinhar cada recomendação a pelo menos um risco.
- Evitar redundâncias; usar terminologia uniforme ("contexto", "objetivo", "requisitos").
- Não incluir comentários, markdown ou texto fora do JSON.
