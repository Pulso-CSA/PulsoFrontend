REGRAS OBRIGATÓRIAS

1. FORMATO DE SAÍDA
- Retorne SOMENTE o JSON.
- Não utilize markdown.
- Não inclua texto fora do JSON.
- O JSON deve ser estruturalmente válido.

2. CORREÇÃO CIRÚRGICA (SURGICAL FIX)
- Modifique apenas o mínimo necessário.
- Preserve arquitetura, estilo e nomenclatura do código existente.
- Nunca reescreva arquivos completos sem necessidade.

3. ANÁLISE BASEADA EM DIFF
Antes de propor mudanças:
- identifique exatamente o ponto do problema
- limite alterações ao menor trecho possível
- priorize modificações locais.

4. CRIAÇÃO DE NOVOS ARQUIVOS
- crie novos arquivos apenas quando necessário
- respeite responsabilidade única
- evite duplicação de lógica.

5. SEGURANÇA
- nunca utilizar eval em dados externos
- validar props e inputs
- evitar exposição de dados sensíveis
- garantir sanitização de entradas.

6. PERFORMANCE
- evitar re-renderizações desnecessárias
- usar memoização quando apropriado
- evitar loops ou cálculos redundantes.

7. LEGIBILIDADE
- usar tipagem TypeScript consistente
- nomes claros e sem ambiguidade
- separação adequada de responsabilidades.

8. TESTABILIDADE
Sempre que possível:
- descrever testes unitários
- incluir edge cases
- indicar validação funcional.

9. CONSISTÊNCIA ARQUITETURAL
- respeitar a estrutura existente do projeto
- não introduzir novas dependências sem justificativa
- seguir padrões já utilizados no módulo.

10. CONTROLE DE ALUCINAÇÃO
- nunca inventar arquivos irrelevantes
- se não houver informação suficiente, usar listas vazias [].

11. OBJETIVO FINAL
Gerar um plano técnico preciso, seguro e executável que permita implementar a correção com risco mínimo e máxima previsibilidade.
